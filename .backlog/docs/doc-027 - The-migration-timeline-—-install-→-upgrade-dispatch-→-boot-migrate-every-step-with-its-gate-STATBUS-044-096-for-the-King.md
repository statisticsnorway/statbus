# The migration timeline тАФ from install to the boot-migrate window

**Question answered** (King, 2026-07-07): *"why do we, at startup, apply migrations blindly? I need a sequence of events here to dig deeper."*

**Short answer first.** Because every upgrade must swap the `./sb` binary, and a Go process cannot hot-swap itself: the upgrade pipeline always ends by exiting so a NEW process тАФ the new binary тАФ takes over (`service.go:4953` `os.Exit(42)` on the service path, `:4959` `syscall.Exec` on the inline install path). The migration delta therefore always lands in the NEXT process's startup window тАФ but since STATBUS-145 the startup window applies only the daemon's own small FLOOR set; the upgrade delta runs exactly once, inside the guarded resume pipeline. That window is **gated, not blind**: a persisted flag names the exact upgrade being resumed, a budget guard counts the pass before any migration runs, parked rows skip it entirely, and the only migrations that can be "pending" are those of a tree someone deliberately put there. "At startup" IS the second half of the dispatched upgrade pipeline тАФ not an independent act.

Each numbered step below is one event, with its trigger and the code that runs. Rev 1 verified against master HEAD 2026-07-07. **Rev 2 (2026-07-12): ┬зB step 6, ┬зC steps 15/16/18, and ┬зD updated to the SHIPPED geometry тАФ STATBUS-145 (floor-bounded boot-migrate, delta-once-in-pipeline), STATBUS-154 (teardown-immune terminals, parkedтЗТin_progress constraint, state-write audit log), STATBUS-159 (displacement-at-claim). All proven end-to-end by the health-park arc, wave 10, run 29171998401.**

## A. Fresh install

1. **Operator runs `install.sh` тЖТ `./sb install`.** `install.Detect` probes the 8-state ladder (`cli/internal/install/state.go`); a fresh box (no `.env.config`) routes to the step-table (`cli/cmd/install.go`).
2. **The step-table's DB steps run in order:** "Seed" тАФ restore the published seed dump if available (`install.go:622`, runner `:1729-1753`) тАФ then "Migrations" тАФ `./sb migrate up --verbose` (`install.go:623`, runner `:1756-1759`). **What "pending" means:** `migrate.HasPending` compares `migrations/*.up.sql` on the working tree against the applied-versions table `db.migration` (`install.go:812-818`). On a fresh box: everything newer than the seed.
3. **Terminal:** schema at the installed tree's HEAD; every applied version recorded in `db.migration`. No flag, no `public.upgrade` row involved.

## B. Upgrade dispatch тАФ the OLD binary's half

4. **Candidate registration:** `./sb upgrade check` fetches releases and registers rows (`service.go:4271`), or `./sb upgrade register <version>` (`:4140`).
5. **Scheduling:** `./sb upgrade schedule <version>` тЖТ `public.upgrade` row `state='scheduled'` + NOTIFY (`:4189`).
6. **Claim тАФ one shared path (STATBUS-159, `claimScheduledUpgrade`, `service.go:4436`):** the daemon's 30s tick (`executeScheduled`) and the install dispatch (`ExecuteUpgradeInline`) both route through the same helper. It first DISPLACES any standing park тАФ a parked row exists precisely to wait for a fix release, so the fix's claim must proceed over it: the parked row's stale service flag is removed, then one transaction moves the parked row to `superseded` (park reason preserved in `error` plus a displacement note, park marker cleared in the same UPDATE) and claims the new row `state='in_progress'` with the `state='scheduled'` guard. A LIVE unparked in-progress row still blocks the claim loudly (the `upgrade_single_in_progress` unique index) тАФ displacement applies to parked rows only. The claim's RETURNING carries `commit_tags, recreate` atomically. Proven live: wave 10, C completed t+66s over a standing park.
7. **Pre-flight (Phase 0, no flag yet):** downgrade / platform / disk / signature checks inside `executeUpgrade` (`:4467`+) тАФ reject cleanly before anything destructive.
8. **FLAG WRITE** (`:4624` `writeUpgradeFlag`): `tmp/upgrade-in-progress.json` + kernel flock. From here a crash is resumable; the flag carries the upgrade id, target commit SHA, phase, and the dying-step fields the budget reads.
9. **Pre-swap destructive steps:** warm-up image pull; read-only window ON (`:4746`); maintenance ON + app/worker/rest stopped (`:4769`+); **DB stopped** (`:4793`); volume snapshot backup (`:4826`); `git fetch` of the target's objects тАФ **no checkout yet** (`:4835`; STATBUS-060: the old binary must never materialize the target's compose/config).
10. **BINARY SWAP:** `replaceBinaryOnDisk` (`:4915`) puts the target `./sb` on disk (old kept as `./sb.old`); flag stamped `Phase=post_swap` (`:4941`).
11. **HANDOFF:** service mode exits 42 (`:4953`; systemd `Restart=always` restarts the unit onto the new binary); inline mode `syscall.Exec`s it (`:4959`). **At this moment ZERO migrations have run**: the DB is stopped, the tree is still the old tree. The old process cannot run them тАФ the migration SQL belongs to the new tree, and the new tree's queries need the new binary.

## C. The NEW binary's boot pass тАФ floor only; the delta lands in the pipeline

Every step below runs on every daemon start (unit start, crash restart, reboot); the flag-gated steps engage only when a flag is present.

12. **Recovery checkout:** a service-held FORWARD flag (post_swap/resuming) тЖТ `git checkout <flag.CommitSHA>` (`service.go:1772-1796`). The working tree is now the target's тАФ its `migrations/` holds exactly the scheduled upgrade's delta. PreSwap flags are gated out (they roll back; a forward checkout would create git/schema skew).
13. **Cheap init:** `./sb config generate` unconditionally (`:1798`); `EnsureDBUp` = `docker compose up -d db` (`:1808`) тАФ revives the DB that step 9 stopped (or a crash killed); connect (`:1812`); advisory lock (`:1819`); LISTEN (`:1844`); `READY=1` (`:1864`) so the heavy work below runs in systemd's ACTIVE phase under WatchdogSec, not the fixed start timeout.
14. **RecoveryBudgetGuard** (`:1895`, definition `:5790`+) тАФ the counting gate, BEFORE any migration: engages only for a service-held forward flag with a dead holder. Takes the flock; if the row is PARKED тЖТ **skip the boot migrate** (alive-idle); else **increments `recovery_attempts`** (the pass is counted before the migrations so a death inside them self-counts), consults the escalation core тАФ same-step-twice or 3-death budget (`recovery_escalation.go:112-132`) тАФ a terminal verdict **PARKS** (never rolls back from this early guard; a deliberate `./sb install` un-parks into the careful routing that can still roll a genuinely-behind box back); otherwise stamps `Step=boot-migrate` on the flag and continues.
15. **BOOT-MIGRATE тАФ floor-bounded (STATBUS-145):** `./sb migrate up --to <DaemonSchemaFloor> --verbose` (`service.go:1926`; the floor constant lives in `cli/internal/migrate/daemon_floor.go:32` with the bump-guard rules). Boot catches the schema up **only to the daemon's own operating floor** тАФ the small set of upgrade-lifecycle migrations the daemon's own SQL needs (today: through the STATBUS-154 parked-invariant constraint and state-log trigger, both on `public.upgrade`). **The upgrade delta does NOT run here.** A flagless boot with migrations pending beyond the floor logs one loud line naming the deferred count (`:2033-2040`) and leaves them for a deliberate upgrade or `./sb install`. Bounded by the 12h ceiling (`watchdog.go:150`, env-overridable, STATBUS-095), covered by an always-ping watchdog ticker.
16. **Failure routing at this site, in order** (`:1938-2018`): TIMEOUT тЖТ the orphaned in-container psql backend is reaped, then falls into the branches below. Service-held flag present тЖТ **defer to `recoverFromFlag`**, the snapshot-restore owner (STATBUS-017) тАФ never refuse-and-loop on a half-applied upgrade. Flagless + deterministic exit 20 тЖТ **one loud actionable report, daemon stays alive-idle** (STATBUS-144). Flagless + transient/unclassified тЖТ refuse + exit; systemd restarts. **Domain shrink (145):** a failure here is a FLOOR-migration failure тАФ our own small lifecycle set тАФ never a broken upgrade delta; a broken delta is above the floor and can only fail inside the guarded pipeline step (18), where observed-state Behind routes to the data-safe rollback.
17. **`recoverFromFlag`** (`:2047` call site) тАФ the routing brain. PreSwap тЖТ one-shot rollback. post_swap тЖТ `resumePostSwap`. Resuming тЖТ observed-state tri-state: at-target тЖТ resume forward; positively-Behind тЖТ data-safe rollback; unreadable тЖТ one bounded backoff-retry, recurrence after a cleared backoff тЖТ rollback.
18. **`resumePostSwap` тЖТ `applyPostSwap` тАФ where the delta actually runs, exactly once:** config-generate тЖТ pull тЖТ db up тЖТ reconnect тЖТ **migrate (the upgrade delta's ONLY consumer since 145** тАФ write-ahead step stamp, 12h ceiling + orphan reap, exit-20/22 classification, observed-state Behind тЖТ data-safe rollback**)** тЖТ start services тЖТ health тЖТ maintenance OFF тЖТ terminal. **Terminal geometry (154/159):** every terminal write goes through `terminalUpdate` тАФ teardown-immune (context.Background + fresh daemon-tagged connection + bounded retry), and exiting as-terminal REQUIRES the write landed, else a loud row-stays-in_progress escalation. A deterministic failure at a post-swap step PARKS instead (`parkForDeterministicFailure`): the row stays `in_progress` with `recovery_parked_at` set тАФ `chk_upgrade_parked_requires_in_progress` makes a parked non-in_progress row unrepresentable тАФ the unit stays alive-idle (14's parked-skip), and the park resolves by deliberate un-park (`./sb install`, re-schedule) or by DISPLACEMENT at the next claim (step 6). Every state / park-marker write on `public.upgrade` is audited by the `upgrade_state_log` trigger (migration 20260711201432) with the writer's `application_name`, backend pid, and statement тАФ the invisible-writer class is closed. Boot-time reconcilers (`completeInProgressUpgrade`, `markCurrentVersionCompleted` `:2922`) are guarded to never steal an `in_progress` or parked row from recovery.

## D. Why the migrations land where they do тАФ and why it is not blind

19. **Why the boot window exists (necessity, not preference):** step 11 always hands off, so SOMETHING must run in the next process before `recoverFromFlag`: the new binary's ~23 `public.upgrade` queries assume the schema its own SQL needs тАФ rc.63 renamed three columns, and a new binary on an old schema dies with SQLSTATE 42703 at its first recovery query (the rc.65 incident). Boot-migrate-to-floor is that self-consistency guard, and no more: binary and schema agree on the daemon's OWN surface before the binary reads what state it is in. Everything else waits for the pipeline.
20. **Why not "blind":** (a) "pending" is never arbitrary тАФ it is exactly the delta of a tree a deliberate act put there: the flag's target checkout (12), install.sh's checkout, or a deploy push; migration files do not appear on their own. (b) The window is counted and bounded by the guard (14): budget, same-step-twice, parked-skip. (c) Its failures are classified by the exit-code contract (`migrate/exit_codes.go:39-44`) and routed per 16, with the snapshot-restore path owning the flagged case. (d) It is time-bounded (12h ceiling) and watchdog-covered. (e) **Delta-once (145):** the upgrade delta has exactly one consumer тАФ the pipeline's guarded migrate step (18) тАФ so no migration beyond the daemon's floor ever runs outside a deliberately dispatched upgrade or install.
21. **The formerly-open arm is CLOSED (145).** Rev 1 recorded one deliberately-open arm: a flagless boot with pending migrations applied them silently. That arm no longer exists. A flagless boot applies only the floor set and logs the deferred-delta count loudly (step 15); the delta applies only through a deliberate act тАФ `./sb upgrade schedule` тЖТ claim тЖТ pipeline, or the operator's `./sb install`. The King's explicit-intent question from Rev 1 is thereby resolved in shipped code: autonomy is preserved (the box never wedges waiting for confirmation тАФ floor migrations keep the daemon self-consistent), and explicitness is enforced exactly where it matters (the delta moves only on deliberate dispatch). Proven end-to-end by the health-park arc's wave-10 full green (run 29171998401): park тЖТ alive-idle тЖТ un-park тЖТ re-park тЖТ displacement-at-claim тЖТ fix completed, data intact, every transition state-logged.
