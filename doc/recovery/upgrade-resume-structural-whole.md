**Status: SHIPPED design record** (converged 2026-05-27, architect; all eight pieces landed — commit refs inline. Two later evolutions, STATBUS-043 2026-07-12: restore-selection passages updated to the STATBUS-039 identity-keyed reality; and the CHANGE-1 forensic tar (`archiveBackup` + checkpoint machinery) was subsequently DELETED outright (STATBUS-112) — the persistent CHANGE-2 snapshot dir is the single backup artifact, so read the tar rows as history). ## The unifying principle (the King's OOM insight)

**Nothing is unkillable** — OOM, reboot, SIGKILL can hit any process at any moment. So "never killed" (the earlier DETACH idea, "make boot-migrate unkillable") was the *wrong* goal. The right invariant, applied **uniformly to every step**:

> **A kill anywhere leaves a clean, resumable state; re-running converges (crash-safe + idempotent-resume); and advancing work is progress-protected (it survives the watchdog while it advances; a genuine hang is caught).**

The root defect was that the exit-42 **resume** ran heavy DB-size-scaled work in systemd's **start** phase under a **fixed** `TimeoutStartSec` — a budget that can never bound DB-size-scaled work, so on NO (32 GB) it was killed every cycle → 40 h loop. The fix is not to dodge the kill; it is to make every step satisfy the invariant so a kill is *survivable and convergent* everywhere.

## How each mechanism meets the invariant

| Step | crash-clean + resumable | progress-protected (liveness) |
|---|---|---|
| **completion** | FIX A: `completed`-UPDATE + flag-removal run *before* forensics (shipped `4d9b1e075`). | — |
| **tar** (archive) | ATOMIC: `.tmp`+fsync+rename; prune reaps `.tmp` orphans (shipped `33a9efc80`). | **CHANGE 1:** `tar --checkpoint=N --checkpoint-action=…` emits per-checkpoint progress → feeds the progress-gated watchdog; a write-hang stops checkpoints → watchdog fires. |
| **rsync** (backup) | **CHANGE 2:** persistent dir, committed by atomic dir-rename (`active`↔`syncing`); rsync idempotent → a killed rsync is *finished* by re-running. | rsync output advances the watchdog. |
| **migrate** (incl. boot-migrate) | each migration file commits in its own transaction; `db.migration` version table makes `migrate up` idempotent/resumable. | the per-migration `[migrate] ▶ applying …` line already fires `WATCHDOG=1` (`migrate.go:644-651`). |
| **the resume** | runs **active-phase** (B1) — the unit is honestly "running while it works". | progress-gated watchdog (#3) catches a true hang. |

The rename state machine (active/syncing) is unchanged and current. The marker design dropped atomic-rename because it assumed atomic-rename needs a *fresh copy* (the shipped tar's `.tmp`→rename), which a persistent-incremental dir precludes. It doesn't: rename the **persistent dir itself** aside and back — incrementality preserved (content survives the rename) AND atomic clean/dirty restored (the dir **name** is the state). `backupDatabase` (`exec.go:261`) keeps the persistent incremental rsync target and commits via one atomic `rename(2)`:
   1. if `pre-upgrade-active/` exists → `rename(active → syncing)` (atomic; **content preserved** = the incremental base). If `active` is absent, a `syncing` left by a killed run is already the base — go straight to rsync; **never delete it** (rsync `--delete` reconciles a partial `syncing` to the source incrementally — that partial IS what you resume from). `active` and `syncing` never coexist from our sequence (architect-confirmed: after the aside-rename `active` is absent until the final rename consumes `syncing`), so there is NO cleanup branch and NO `rm`; an unexpected coexistence makes the aside-rename fail loudly = fail-fast, which is correct.
   2. `rsync --delete <db> → pre-upgrade-syncing/` (incremental, idempotent-resumable).
   3. fsync the synced dir before the commit (mirrors the shipped ATOMIC tar's fsync — durable across power-loss).
   4. `rename(syncing → active)` — `active` is guaranteed ABSENT here, so this is a clean atomic commit. The backup is complete **iff** it is named `pre-upgrade-active/`.
   - **A partial is structurally invisible (identity-keyed, STATBUS-039):** `restoreDatabase` reads ONLY the recorded `flag.BackupPath`/`row.backup_path`, and a dir named `syncing` is never recorded — so a partial backup cannot be restored by construction, with no marker-check to forget and no recency scan to guess wrong. `pruneBackups` → near-noop on the managed dirs; historical retention in the tars (`pruneArchives`, keep 3).
   - **Why rename dominates the marker:** (a) one atomic metadata op commits "clean" — the SAME pattern as the shipped ATOMIC tar, one mental model across both backup mechanisms; (b) the state IS the name — no second artifact (dir + marker) that can contradict; (c) **fail-safe by construction** — consumers look only for the clean name, so a partial can never be trusted even by forgetful consumer code (the marker required every reader to remember the check); (d) simpler durability ordering (fsync data → rename) than the marker's (fsync data → fsync marker → fsync parent).
   - **Idempotent resume (no deletion ever):** a kill mid-sync leaves `pre-upgrade-syncing/` (partial); re-run rsyncs `--delete` straight into it (reconciling the partial to the source) and commits via the final rename. The dir is never `rm`'d — that would discard the incremental base and force a full re-sync.
   - **Single-dir safety (unchanged argument):** `rename(active→syncing)` hides the last-good during the sync window, but `backupDatabase` runs at pre-swap, **before any destructive step** (migrate `:3726`/swap `:3424` later). A kill *during* backup ⇒ nothing destructive happened ⇒ live DB intact at pre-N ⇒ **no rollback artifact needed**; by the time a later step could fail, `pre-upgrade-active` is re-committed. The hidden-last-good window coincides exactly with the nothing-destructive-yet window. Safe.

(fsync the parent after the rename is optional: a lost rename just leaves `syncing` = redo = acceptable.)
